# RN 对齐 Android

当任务是把 Android 原生页面、组件、弹窗、底部栏、信息层或交互行为严格对齐到已有 RN 实现时，使用这个技能。

## 核心规则

- RN 是唯一视觉和交互基准，包括布局、间距、字体、图标、动效、文案、层级、显隐和点击路径。
- 先看 RN，再改 Android。
- 先定位根因，再修改代码，禁止猜改。
- 只做最小、直接、稳定的修改，不扩大改动面。
- Android 可以不用机械照搬 RN 内部实现，但必须保证目的相同、结果相同、用户可见行为相同。
- 不能因为“更像 Android”就替换 RN 的表现。

## 修改前必须先说清楚

必须先明确：

1. RN 目标表现：
   - 布局结构
   - 间距和字体层级
   - 安全区行为
   - 弹窗、蒙层、底部弹层的覆盖范围
   - 状态切换和交互路径

2. Android 当前表现：
   - 差异点是什么
   - 在什么场景下出现
   - 如何复现

3. 差异最可能落在哪一层：
   - 主题 token 或缩放
   - safe area / inset
   - 布局测量
   - 状态归属
   - overlay host / z-order
   - repository / 请求时序
   - 初始化时机

在没有把问题挂到明确责任层和证据链之前，不要动代码。

## 标准迁移流程

1. 先读 RN 对应页面或组件。
2. 再定位 Android 对应文件：
   - Screen 组装层
   - 可复用 section / card
   - theme token / scale
   - safe-area 容器和 header
   - overlay host / bottom sheet
   - 如果差异和数据或时序有关，再看 ViewModel / Repository / Bootstrap
3. 一块一块比对可见输出和交互路径，不要整页凭感觉重写。
4. 优先找这类根因：
   - token 用错
   - 容器层级不对
   - inset 重复叠加
   - 缺少运行时测量
   - 状态源不对
   - 初始化时机不对
5. 在最窄的责任层做最小修改。
6. 按 RN 回归验证原问题和相邻路径。

## 自适应样式规则

- 只要某个值本质上来自主题、缩放、安全区、内容实际高度、宽高比，就不要在页面里写死 magic number。
- 优先复用项目里的 theme token、scale、safe-area 容器、统一图片层，不要把裸 `dp` 散落在各个页面。
- RN 如果表达的是比例关系，不要在 Android 上冻结成一个固定值。
- 底部保留区如果依赖真实信息层高度，必须由测量结果驱动，不要写死常量。
- 字体、行高、间距要跟随布局尺寸关系自适应，不能在横屏、小屏、长屏下整体失真。
- 背景、渐变、头图、装饰图从真正的屏幕顶部开始画；安全区只影响内容起点，不影响背景起点。

## Safe Area 与弹层规则

- `SafeScreen` 语义必须和 RN 一致：背景可全屏绘制，内容只在 RN 避让时才避让。
- 页面、Header、内容层不能重复叠加顶部 padding。
- 问号弹窗、奖励弹窗、全屏蒙层的覆盖范围必须与 RN 一致。RN 是全屏，就不能做成半屏 dialog。
- 点击拦截区、暂停热区、底部禁点区，优先由实际 UI 层级和测量结果驱动，不要依赖过期的固定高度。
- 底部手势区或三键区背景要和 RN 的 tab bar / sheet 视觉连起来，不能断层。

## 状态、数据与初始化规则

- 不要把应用级初始化塞进 Home 或任何业务页面。
- 初始化只允许三层：
  - `Core SDK Bootstrapper`
  - `App Bootstrap`
  - `Scene Warmup`
- 如果页面差异本质上是 bootstrap 时机、token 状态、游客态、配置预热问题，就修归属和时机，不要拿 UI 打补丁。
- 页面只负责组装 UI 和派发事件，不要在 Composable 里拼请求、兜 token、兜缓存。

## 可优于 RN 的边界

如果 RN 为了历史包袱、平台妥协或性能问题，用了脆弱实现，Android 可以换更稳的方案，但必须先说明：

- RN 现在怎么做
- RN 想达成的目的是什么
- RN 方案在 Android 上为什么不合适
- Android 替代方案为什么更稳、更清晰或性能更好
- 为什么用户可见结果仍然严格对齐

不能先改结果，再事后解释。

## 明确禁止

- 没看 RN 就直接改 Android
- 因为一个局部差异就整页重写
- 用“猜出来的 Android 风格”替换 RN 样式
- 用大段 `if/else`、fallback、延时、强刷、重复状态掩盖问题
- 把该自适应的间距、高度、弹层范围、保留区写死
- 让页面持有 bootstrap、缓存、token、SDK 初始化职责
- 把“差不多”当成已经 1:1 还原

## 验收清单

结束前必须对照 RN 验证：

- 结构和 section 顺序
- 间距、圆角、字体、颜色
- 主文案和副文案
- safe-area 起点
- 蒙层范围和遮罩覆盖
- 滚动、滑动、点击路径
- 空态、loading、错误态、正常态
- 如果 RN 支持主题化，要验证主题切换
- 如果场景支持横屏，要验证横竖屏

输出结论时必须说明：

- 根因是什么
- 最小修改落在哪一层
- 为什么这个改动仍然保持 RN 对齐
- 已验证什么
- 还有什么没验证
