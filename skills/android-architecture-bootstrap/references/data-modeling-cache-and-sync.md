# 数据建模、缓存与同步

## 目标

生产级数据层的关键不是能请求成功，而是：

- 数据模型清晰
- 真相源明确
- 缓存策略稳定
- 同步行为可预测
- 异常语义统一

## 模型分层建议

至少区分下面几层：

- `Dto`：接口模型
- `Local Model` / `Entity`：本地存储模型
- `Domain Model`：业务实体
- `Ui Model`：页面展示模型

不要让接口字段直接穿透到 UI。

## Repository 规则

- 接口定义在稳定层
- 实现放 data 层或 feature data 层
- Repository 对外暴露业务语义，不泄露 Retrofit / Room / SDK 细节
- 多数据源合并在 Repository 内完成

## 单一事实源

设计前先回答：

- 这个数据最终以谁为准
- 内存缓存只是加速还是主事实源
- 本地数据库和接口冲突时谁覆盖谁
- 页面刷新后依赖的是数据库回流还是网络直出

## 缓存策略建议

常见模式：

### 在线优先

适合：

- 数据实时性要求高
- 本地只是短暂缓存

### 本地优先

适合：

- 列表、详情、历史、离线浏览
- 冷启动需要秒开

### 同步驱动

适合：

- 文件、消息、下载、上传、配置、任务系统

## 同步设计问题

必须提前定义：

- 首次同步时机
- 增量同步条件
- 全量重拉条件
- 冲突解决策略
- 失败重试和退避策略
- WorkManager 唯一任务名和 KEEP / REPLACE 策略
- 幂等键和去重规则

## 错误语义

不要把底层异常直接扔给 UI。

建议统一成：

- 用户可理解的业务错误
- 需要重试的系统错误
- 需要登录或权限处理的错误
- 可静默上报的技术错误

## 数据层目录建议

```text
feature-xxx-data/
  remote/
  local/
  repository/
  datasource/
  mapper/
  model/
  worker/
```

或在已有 data 层中按业务域切子目录。

## 数据库和迁移建议

- 任何结构化数据都要明确版本迁移策略
- 索引和查询热点要提前设计
- 大表先考虑分页和归档
- 迁移脚本要可测试

## 不推荐做法

- ViewModel 同时依赖 API、DAO、DataStore
- 缓存策略完全靠调用方自行猜测
- Worker 重复 enqueue 却没有唯一名和策略
- 用默认值把字段缺失悄悄吞掉，最后污染真相源
