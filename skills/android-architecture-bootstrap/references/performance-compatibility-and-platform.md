# 性能、兼容与平台能力层

## 性能设计不是后补项

架构阶段就要明确：

- 冷启动预算
- 首屏预算
- 滚动和输入响应预算
- 包体积预算
- 后台任务和同步预算

## 启动性能建议

分三层看：

- 必须同步初始化
- 可延后初始化
- 首次真正使用时初始化

规则：

- Application 不做重 IO 和重计算
- 第三方 SDK 按必要性分层初始化
- 启动链路里的日志、埋点、配置读取要算预算

## 性能基建建议

至少规划：

- `Baseline Profile`
- `Macrobenchmark`
- 启动 trace
- Crash / ANR 监控
- 主线程阻塞监控
- 包体大小趋势监控

## 列表与渲染性能

- 大列表优先分页和懒加载
- 稳定 key
- 重映射和格式化放后台线程
- 避免高频状态导致整页反复重组

## 平台能力层建议

当项目有下面能力时，优先抽 `platform`：

- 权限申请和系统设置跳转
- 通知和前台服务
- 下载、上传、文件访问、媒体采集
- WebView 和 JS Bridge
- 支付
- 推送
- 定位 / 蓝牙 / NFC / 生物识别

目标：

- 业务代码不直接散落系统 API 细节
- Android 版本差异收口到平台能力层
- 测试和 mock 更容易

## 兼容矩阵必须提前定义

至少覆盖：

- minSdk 到 targetSdk 的关键行为差异
- 手机 / 平板 / 折叠屏 / 横屏 / 分屏
- 深色模式
- 字体缩放
- RTL
- 多语言长文案
- 厂商 ROM 特殊行为
- 权限、通知、后台限制差异

## 自适配规则

- 优先用窗口尺寸和布局约束，不靠设备型号判断
- 大屏先调布局结构和容器宽度，不先无限放大字号和间距
- IME、系统栏、安全区要进入共享处理逻辑

## 后台任务建议

- 唯一任务使用唯一名
- 明确网络、充电、存储等约束
- 任务实现要幂等
- 重试要有策略，不要无限试错

## 不推荐做法

- 卡顿了先乱加缓存
- 兼容出了问题先写设备白名单 if else
- 平台能力到处散落在 feature 页面和 Fragment 里
