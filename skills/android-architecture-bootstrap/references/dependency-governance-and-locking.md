# 依赖治理与锁文件

## 目标

生产级项目的依赖治理要解决：

- 仓库源失控
- 版本来源分散
- 同库不同版本混入
- 私有仓库和本地调试能力不透明
- CVE 修复没有落点

## 推荐基线

- `RepositoriesMode.FAIL_ON_PROJECT_REPOS`
- `version catalog`
- BOM 或约束统一版本
- dependency locking
- 依赖准入标准
- 私有仓库和本地替换策略显式化

## 仓库治理

推荐：

- 仓库统一在 settings 管理
- 模块内禁止自定义仓库
- 私有仓库需要明确认证来源
- 本地调试仓库只能通过显式开关启用

## 版本治理

- 基础依赖统一从 version catalog 获取
- 同类库优先使用 BOM
- 风险依赖通过 constraints 或 force 收敛
- 升级策略要有 owner

## dependency locking 建议

适合：

- 团队多人协作
- 构建可复现要求高
- CI 和本地结果要稳定一致

规则：

- 默认锁全部 configuration 或至少锁主配置
- 依赖更新走明确流程，不允许隐式漂移
- 锁文件变更应进入 review

## 私有仓库与本地 SDK 替换

常见需求：

- 私有 Maven
- GitHub Packages
- 本地 SDK 调试

规则：

- 认证信息来自 `user.properties`、环境变量或 CI secret
- 本地替换要有显式开关，禁止默认开启
- 替换规则要可发现、可关闭、可回归

## 依赖准入问题

新增库前至少回答：

- 这个能力是否已有等价库
- 是否能用官方或主流方案替代
- 对包体、方法数、启动、权限、混淆的影响是什么
- 是否需要长期跟版本或安全更新

## 安全与补丁治理

- 高风险基础库要有 CVE 响应策略
- 临时强制版本要记录原因
- 安全补丁不能只改一个模块，必须看全局依赖树

## 常见错误

- 模块自己加仓库
- 依赖版本散落在各个 build 文件
- 本地 SDK 替换没有开关与回归路径
- 线上构建和本地构建依赖树不一致
