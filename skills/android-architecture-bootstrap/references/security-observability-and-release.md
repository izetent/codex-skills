# 安全、可观测性与发布治理

## 安全边界

架构设计时就应明确：

- 哪些数据属于敏感数据
- 哪些模块允许访问敏感数据
- 日志脱敏规则
- 本地存储加密边界
- WebView、文件分享、导出、深链的安全校验边界

## 安全建议

- token、session、手机号、邮箱、证件号等不明文落日志
- 敏感数据优先走加密存储
- 外部输入全部做边界校验
- 导出的 Activity / Provider / File 场景必须显式审查
- WebView 功能和 bridge 能力收口，不让业务随意直连

## 可观测性建议

至少规划：

- 崩溃监控
- ANR 监控
- 性能指标
- 关键业务埋点
- Trace 或链路标识
- 关键后台任务运行状态

日志规则：

- 能帮助定位问题
- 不能泄露敏感信息
- 不能大面积重复刷日志

## 环境与配置治理

提前定义：

- 环境配置来源
- 构建时注入和运行时注入边界
- 密钥和敏感配置如何管理
- debug / qa / staging / prod 差异如何表达

## Feature Flag 与灰度

推荐能力：

- 本地开关
- 远程开关
- 实验分桶
- kill switch

规则：

- feature flag 只控制能力开关，不替代架构边界
- 高风险能力必须支持快速熔断

## 发布治理建议

至少设计：

- 发布渠道
- 发布前门禁
- 版本说明和变更记录生成策略
- 映射文件与符号文件归档
- 灰度、回滚和紧急止血机制

## 监控闭环

发布后要能回答：

- 哪个版本 crash / ANR 异常升高
- 哪条关键链路性能退化
- 哪个开关或实验引入了回归
- 哪个后台任务失败率异常

## 不推荐做法

- 安全只靠上线前人工过一遍
- 没有灰度能力却直接全量发布高风险改动
- 线上问题只能看用户截图，缺少监控与链路信息
