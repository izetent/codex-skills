# 测试、Review 与协作规范

## 适用场景

- 补测试
- 做代码评审
- 准备 PR
- 判断修改后应该验证什么

## 测试规则

- 测试加在真正拥有该逻辑的模块。
- ViewModel、UseCase 一般要有单测覆盖。
- Repository 重点覆盖映射、持久化、边界条件、异常路径。
- 关键 UI 状态尽量有 screen test 或 component test。
- 优先复用共享测试模块和测试工具，不要每个模块各自拼一套。

建议最少覆盖：

- 新增 UseCase：成功路径 + 失败路径
- 状态切换逻辑：初始态 + 动作后状态
- Bug 修复：原始复现路径对应的测试
- 布局类修改：至少补一条关键组件或页面断言

## 验证规则

至少验证：

- 主用户路径
- 一个邻近回归路径
- 一个生命周期或状态恢复路径
- 若页面存在空态、加载态、错误态，至少抽查相关分支

如果是布局类修改，至少补看：

- 竖屏
- 横屏
- 至少一个更宽的设备形态
- 长文案或本地化文案场景

## Review 规则

如果用户要 Review，优先检查：

- 行为回归
- 依赖方向错误
- 状态归属混乱
- 生命周期泄漏
- 重复真相源
- 测试缺失

结论要短，问题优先，摘要靠后。

建议 Review 输出结构：

1. 先列问题
2. 再列开放疑问
3. 最后补一行整体判断

## 协作规则

- 提交保持小步、聚焦、可回滚
- Bug 修复不要混入无关清理
- 架构调整要说明为什么需要改边界
- UI 修改要说明可见影响和验证路径
- 模块变更要说明所有权和依赖方向变化

## 提交规范

提交前缀统一使用：

- `feat`
- `fix`
- `docs`
- `style`
- `refactor`
- `perf`
- `test`
- `chore`
- `revert`
- `build`

提交信息建议中文，直接描述结果：

- `fix: 修复设备中心页面首次进入空白问题`
- `refactor: 拆分相册页筛选区域状态归属`
- `test: 补充下载列表排序用例`

不推荐：

- `update`
- `fix bug`
- `try`
- `final`

## 最终说明模板

任务收口时，至少说明：

- 问题或目标
- 根因或架构判断
- 最小改动摘要
- 实际验证内容
- 剩余风险或未验证项

## PR 描述建议

- 问题背景
- 根因或改动动机
- 修改范围
- 验证步骤
- UI 变化截图或录屏
- 风险说明
