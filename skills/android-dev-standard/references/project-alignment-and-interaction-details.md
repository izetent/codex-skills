# 项目对齐与交互细节规范

## 适用场景

- 页面开发
- 页面重构
- 输入框、表单、搜索页
- 列表页、筛选页、详情页
- 需要补齐交互细节的场景

## 项目对齐原则

skill 不是强制覆盖项目现状的模板。

如果项目已经有明确约定，优先对齐现有项目，而不是把页面硬改成另一套写法。至少先判断：

- 当前项目是 Compose 还是 XML 为主
- 当前项目是否已有设计系统
- 当前项目的导航方式是什么
- 当前项目的 DI 体系是什么
- 当前项目的模块命名和目录结构是什么
- 当前项目是否已经有统一输入框、列表、弹窗、搜索栏组件

## 页面拆分时的项目对齐

- 如果项目已有 `Route / Screen / ViewModel` 模式，沿用它。
- 如果项目是 `Fragment + XML + ViewBinding` 体系，不强行拆成全 Compose。
- 如果项目已有统一组件库，页面拆分优先复用现有组件，不另起炉灶。
- 如果项目已有状态事件约定，例如 `UiState + Event`、`State + Effect`、`Command`，优先跟现有风格保持一致。

## 样式简洁化原则

所谓“简洁化”不是简单删样式，而是：

- 去掉重复层级
- 去掉不必要的装饰
- 去掉无意义的多套间距和字号
- 收敛为设计系统和主题 token
- 保留页面原有视觉语言

不推荐为了“简洁”把已有设计系统组件替换成原生默认组件。

## 输入框与表单细节

只要页面里有输入框，就至少检查下面这些点：

- 是否设置正确的输入类型或 `keyboardType`
- 是否设置正确的 `imeAction`
- 多输入框表单是否支持 `Next` / `Done`
- 点击提交、搜索、完成后是否应 clear focus
- 键盘弹起后输入框和提交按钮是否被遮挡
- 页面切换、弹窗关闭、提交成功后键盘是否应该收起

## Compose 输入框建议

推荐处理：

```kotlin
val focusManager = LocalFocusManager.current
val keyboardController = LocalSoftwareKeyboardController.current

TextField(
    value = keyword,
    onValueChange = onKeywordChange,
    keyboardOptions = KeyboardOptions(
        imeAction = ImeAction.Search
    ),
    keyboardActions = KeyboardActions(
        onSearch = {
            focusManager.clearFocus()
            keyboardController?.hide()
            onSearch()
        }
    )
)
```

规则：

- 提交动作明确时，优先在 `onDone`、`onSearch`、`onSend` 等 IME action 中 clear focus 并尝试隐藏键盘。
- 隐藏键盘属于副作用，应放在事件回调中，不要在重组过程中直接调用。
- 搜索页、筛选页、表单页如果用户已经开始滚动内容，通常应视为用户离开输入状态，可以收起键盘并清掉焦点。
- 但如果页面交互明显依赖持续输入，不要在每次轻微滚动时强制收键盘。

## View / XML 输入框建议

- `EditText` 明确声明 `inputType`
- 需要时配置 `imeOptions`
- 焦点变化时，如交互上确实意味着离开编辑态，可以手动收起键盘
- 通过 `SoftwareKeyboardControllerCompat` 或等效官方方式隐藏键盘，不使用零散兼容写法拼接

## 键盘关闭规则

下面场景通常应该考虑关闭键盘：

- 用户点击搜索 / 完成 / 提交
- 用户切换页面或弹层
- 用户点到明确的非输入区域，且当前页面交互期望离开编辑态
- 用户开始滚动结果列表，且输入目的已完成

下面场景不要粗暴关闭：

- 用户仍在多字段表单连续输入
- 页面只是轻微滚动但输入还在进行
- 用户切换输入框焦点

## 滚动与键盘联动

推荐在这些页面补齐“滚动关闭键盘”：

- 搜索页
- 筛选页
- 长表单
- 评论输入后查看结果列表的页面

实现目标：

- 开始滚动结果区时，收起键盘
- 结果区获得可用空间
- 焦点离开输入框

但前提是该行为符合页面预期，不能全局一刀切。

## 点击空白处关闭键盘

如果页面是典型表单、搜索、筛选界面，通常建议支持：

- 点击空白区域关闭键盘
- 点击列表内容关闭键盘

如果页面本身可点击区域很多，需避免和主要点击操作冲突。

## 焦点管理

- 大屏、平板、硬件键盘场景下，要考虑 focus 顺序。
- 表单输入建议明确 `Next -> Next -> Done` 流。
- 不要让隐藏组件或无意义装饰元素参与焦点链。
- 弹窗、bottom sheet 打开后，要确认焦点是否应移动到首个可交互组件。

## 常见漏项清单

- 搜索框回车后没有触发搜索
- 提交后键盘没收起
- 键盘挡住底部按钮
- 列表滚动后键盘仍占空间
- 页面退出后输入焦点没清理
- 空白点击关闭键盘和列表点击事件冲突
- 多输入框的焦点切换不连贯
