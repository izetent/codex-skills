# 性能、稳定性与安全规范

## 适用场景

- 优化启动、列表、滚动、输入响应
- 排查卡顿、ANR、内存、重复任务
- 处理后台任务、离线同步、缓存
- 审查敏感数据与异常处理

## 性能规则

- UI 线程只做轻量工作。
- 重映射、重排序、大列表组装放后台线程。
- 长列表优先分页、懒加载、稳定 key。
- 频繁重组或频繁刷新时，先查状态源和订阅链路，再查布局本身。
- 不做无证据的过早优化；先找到慢点，再下手。

## Compose 性能规则

- 不在 Composable 中直接启动不受控协程
- 不在组合阶段做重 IO 或重计算
- 大列表项保持参数稳定
- 图片、缩略图、格式化等高频逻辑尽量提前处理

## Worker 与后台任务规则

- 唯一任务要用唯一名
- 重复触发场景必须显式说明 KEEP / REPLACE 策略
- 后台任务尽量幂等
- 必须明确约束条件：网络、充电、存储、前后台等
- 出错后是重试、终止，还是等待下一次调度，必须有策略

## 稳定性规则

- 异常处理要分层，不要每层都吞
- 错误日志要能反查链路，但不要输出敏感信息
- 对外部数据一律做边界校验
- 生命周期敏感对象不能无边界持有页面引用
- 任何 fallback 都要解释为什么合理

## 安全规则

- token、session、隐私信息不打印日志
- 敏感信息优先加密存储
- WebView、文件、分享、导出等场景要检查边界权限
- 不把内部错误细节直接暴露到用户文案

## 排查清单

- 卡顿发生在首屏、列表、图片、还是状态切换
- 是否有主线程阻塞
- 是否有重复订阅导致重复渲染
- 是否有重复 Worker 或重复请求
- 是否有大对象长期驻留
- 是否有敏感信息落日志或明文落盘

## 明确避免

- 发现卡顿就先乱加缓存
- 发现状态抖动就先强制去重或延时
- 发现后台任务重复就先加布尔值挡住
- 发现错误就统一 try/catch 后吞掉
