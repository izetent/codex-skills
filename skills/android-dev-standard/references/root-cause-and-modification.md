# 根因定位与修改规范

## 适用场景

- 修复 Bug
- 处理回归
- 排查 crash、ANR、白屏、重复请求、状态错乱、时序问题
- 判断当前是否已经具备直接修改条件

## 最高优先级原则

先定位根因，再做最小修改。

没有说清楚下面这些内容前，不直接改代码：

- 现象是什么
- 在什么条件下触发
- 如何复现
- 影响范围有多大
- 真正的状态来源或责任代码在哪里
- 证据是什么

## 排查清单

1. 先定义现象
   - 用户实际看到了什么
   - 是必现、偶现，还是和机型、账号、网络有关

2. 再梳理链路
   - 页面入口
   - 用户操作序列
   - 生命周期节点
   - 数据加载链路
   - 状态更新链路

3. 再判断影响范围
   - 单页面
   - 单模块
   - 单构建变体
   - 特定设备或系统版本
   - 全局状态

4. 再收集证据
   - 日志
   - 断点
   - 调用链
   - 堆栈
   - Flow / LiveData 发射链路
   - Repository 返回值
   - Room / DataStore 实际数据
   - 接口返回
   - Worker 调度状态

5. 区分根因和表现
   - 白屏是表现，不是根因
   - 重复请求是表现，不是根因
   - 缺少状态发射、重复订阅、重复 enqueue 才可能是根因

## 修改规则

- 只改真正拥有该问题的那段代码。
- 能修状态源，不修 UI 兜底。
- 能修生命周期或时序，不加随意延时。
- 能修类型建模，不靠 `Any`、默认值、过度判空掩盖。
- 能删错逻辑，就不要叠更多分支。

## 明确禁止

- 还没定位清楚就先改
- 用大量 `if/else` 遮住异常状态
- 不分场景直接补默认值
- 用重试、延时、强刷、双缓存压住问题
- 问题边界没搞清楚时顺手大重构
- 只是“不报错了”就认定问题修好了

## 可接受的临时排查手段

只有在回答具体未知点时才加临时日志：

- 某方法是否真的被调用
- 某条 Flow 是否真的发射
- 当前线程或生命周期状态是什么
- Repository 拿到的是旧数据还是新数据
- Worker 是否被重复调度

验证完成后，临时日志应移除；除非保留它确实有长期诊断价值。

## 动手前必须明确

- 当前根因假设
- 支撑该假设的证据
- 还有哪些备选解释没有完全排除
- 为什么当前方案是最小必要修改

## 修改后至少验证

- 原始复现路径
- 一个邻近回归路径
- 一个状态恢复或生命周期路径
- 若页面有空态、加载态、错误态，至少抽查一个相关分支
